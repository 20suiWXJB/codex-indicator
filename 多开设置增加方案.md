阶段 1:设置模型(前端 src/settingsModel.ts)

1. AppSettings 接口新增 3 个字段(instanceActiveWindowMinutes 已存在):
   /** session 文件多少秒内有写入才算"运行中"，超时降级为空闲 */
   sessionRunningTtlSeconds: number;
   /** 展开面板顶部是否显示活跃实例列表 */
   showInstanceList: boolean;
   /** 最近事件行是否加"项目名:"前缀 */
   eventInstancePrefix: boolean;
2. DEFAULT_SETTINGS 加 sessionRunningTtlSeconds: 120, showInstanceList: true, eventInstancePrefix: true。
3. SETTING_SECTIONS:在 sources 后新增分组:
   {
   id: "instances",
   title: "多实例",
   items: [
   { key: "instanceActiveWindowMinutes", ...原样从 sources 挪入... },
   { key: "sessionRunningTtlSeconds", title: "运行判定窗口",
   input: "number", min: 30, max: 1800, step: 30, unit: "秒" },
   { key: "showInstanceList", title: "显示实例列表", input: "toggle" },
   { key: "eventInstancePrefix", title: "事件行实例前缀", input: "toggle" },
   ],
   }
3. 并从 sources.items 中删除 instanceActiveWindowMinutes 一项。导航栏由 SETTING_SECTIONS 自动生成,无需另改。
4. normalizeSettings 加三行:numberValue(source.sessionRunningTtlSeconds, 120, 30, 1800) 和两个 boolValue。
5. formatSettingValue 的 switch 加 case "sessionRunningTtlSeconds": return \${value}秒``;两个 toggle 走默认的"开启/关闭"分支,不用改。

阶段 2:Rust 后端(src-tauri/src/lib.rs)——只为"运行判定窗口"
- cached_session_snapshot(path, modified, size)(lib.rs:625)→ 加 running_ttl,透传给两处 snapshot_from_cache 调用;
- active_codex_session_instances(sessions_root, active_ttl)(lib.rs:590)→ 加 running_ttl;
- read_effective_statuses_from_dirs_with_ttl(lib.rs:506)→ 签名改为 (root, sessions_root, active_ttl, running_ttl);
- 常量 SESSION_RUNNING_TTL(lib.rs:20)改名为 DEFAULT_SESSION_RUNNING_TTL,只在默认路径使用。
3. 公开入口保持兼容:read_effective_statuses_from_dirs / read_effective_status_from_dirs(lib.rs:498-504)继续两参数,内部传两个默认 TTL——现有测试和回退路径零破坏。
4. 命令接线:get_status / get_statuses(lib.rs:1304-1322)在 active_window_ttl(&settings) 旁边加一个 running_ttl(&settings) 辅助函数(Duration::from_secs(clamp(30,1800)))一并传入。status_scan_roots() 已返回 settings,不用动。

为什么 bridge(hooks/agent-status-bridge.ps1)不动:运行判定只作用于 session 文件推断链路,bridge 状态有自己的 TTL 语义,不在本次范围。

阶段 3:设置页(src/settingsPage.ts)

只需改 readFormSettings()(settingsPage.ts:162)加三行:
sessionRunningTtlSeconds: value("sessionRunningTtlSeconds"),
showInstanceList: checkbox("showInstanceList"),
eventInstancePrefix: checkbox("eventInstancePrefix"),
渲染、导航、当前值回显全部由 SETTING_SECTIONS 驱动,自动生效——这正是当初把设置做成数据驱动的收益。

阶段 4:主面板消费开关(src/mainPanel.ts)

1. 实例列表开关:renderInstances()(mainPanel.ts:138)入口处判断——settings.showInstanceList === false 时清空 .instances 并 hidden = true 直接返回(隐藏整个 <ol> 而不是渲染空态,否则会留下"暂无实例"占位噪音)。
2. 事件前缀开关:事件渲染处(mainPanel.ts:183-185)把 eventInstanceLabel(event) 的使用改为 settings.eventInstancePrefix ? eventInstanceLabel(event) : ""。
3. 设置热更新:已有 SETTINGS_CHANGED 监听(mainPanel.ts:338)会重新加载设置并刷新,两个开关保存后即时生效,无需新代码;但需确认 applySettings 后触发了一次 renderStatus——若没有,补一次调用。
4. 渲染指纹:renderStatus 的去重 key(mainPanel.ts:102 附近)需把两个开关值拼进去,否则"只改开关、状态没变"时跳过重绘,开关看起来不生效。

提交 1: 阶段 1+3+5(前端部分) —— 分组挪位 + 两个纯前端开关,含运行窗口的 UI(后端未接线前保存了也无害,serde 会带默认值)

提交 3: 阶段 4 —— 主面板消费开关 + README

每个提交独立可编译、可测试。风险点只有一个:阶段 2 的函数签名改动波及 snapshot_from_cache 的所有调用方,编译器会兜底查漏,不会有静默遗漏。
